# Hardware Deployment and Pin-Configuration Guide

## 1. Functional Overview
This document specifies the physical serial port mapping, transceiver wiring schematics, and Ethernet TCP/UDP socket configurations required to deploy the UNIVAC Replacement Bridge Matrix software onto the edge computing platform. Strict adherence to these physical and network topology parameters is mandatory to guarantee hard-deterministic communication loops across the maritime navigation bus.

## 3. Transceiver Pin-Out and Copper Wiring Architecture
All critical sensor feeds operate over balanced, differential RS-422 configurations to guarantee total immunity against high-power electromagnetic interference generated by naval radar, antennas, and machinery spaces.

### A. RS-422 DB9 Terminal Pin Configuration (Ports 0, 1, 2)


5 4 3 2 1
○ ○ ○ ○ ○
○ ○ ○ ○
9 8 7 6

* **Pin 1**: TXD- (Transmit Differential Negative)
* **Pin 2**: TXD+ (Transmit Differential Positive)
* **Pin 3**: RXD+ (Receive Differential Positive) — *Connect to Sensor TX+*
* **Pin 4**: RXD- (Receive Differential Negative) — *Connect to Sensor TX-*
* **Pin 5**: GND (Isolated Signal Ground) — *Mandatory to match voltage baselines*

### B. Structural Shielding Rules
1. **Cable Specification**: Twisted-pair, 24 AWG shielded copper cabling (minimum 2 twists per inch).
2. **Ground Drainage**: Connect the outer braided shield to the ship’s hull chassis ground at the **edge computer terminal plate only**. The sensor end of the shield must float to prevent destructive ground-loop current paths.


## 4. Network Socket Topology (Ethernet TCP/UDP Matrix)
Local network communications route over a dedicated, ruggedized Gigabit Ethernet switch. The bridge matrix isolates command input pipes from active outbound status telemetries by splitting them across distinct socket layers.

## 5. Deployment Pre-Flight Checklist
Before booting `main.py` inside the runtime ecosystem, the platform engineer must execute the following network validation commands from the local command terminal line interface:

### A. Serial Interface Device Verification (Linux Environment)
Ensure the opto-isolated USB interfaces are mapped correctly to the logical software definitions:

ls -l /dev/ttyUSB*

*Expected Output: Verified presence of nodes `ttyUSB0` through `ttyUSB3` with active root/dialout execution privileges.*

### B. Ethernet Interface Binding Verification
Test end-to-end network loop accessibility to the targeted physical motor drive PLC system:

ping -c 3 192.168.1.50

*Expected Output: 0% packet loss response, indicating valid physical copper link across the local switches.*

### C. Socket Allocation Port Check
Verify that port `7000` is completely clear of binding locks from legacy software applications:

netstat -tuln | grep 7000

*Expected Output: Empty return window, confirming port 7000 is open and clear for immediate TCP connection listener registration.*
